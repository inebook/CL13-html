3ファイルの管理：パス方式と ID 方式

ファイルを「どうやって 1 つに特定するか」は、保存先によって考え方が違います。 パソコンや USB（ローカル）と クラウドドライブ（Google ドライブなど）で、 その仕組みを比べてみましょう。

📁 パス方式（ローカル）

ファイルを 「置いてある場所」で特定します。フォルダの階層を上から順にたどった道順が「パス」です。

C:\Users\meijo\卒論\report.docx
  • フォルダを移動・名前変更すると パスが変わる。
  • その結果、ショートカットや文書内のリンクが 切れることがある。
  • 同じ場所に同じ名前のファイルは置けない。

☁ ID 方式（クラウド）

ファイル 1 つ 1 つに 固有の ID（番号）が割り振られ、その ID で特定します。共有 URL の中に ID が入っています。

https://drive.google.com/file/d/1AbC...xyz/view
  • フォルダを移動しても名前を変えても ID は変わらない。
  • だから共有リンクは 切れにくい。
  • 同じ名前のファイルが複数あっても、ID が違えば別物として区別できる。
🔑 ここがポイント: 「場所（パス）で管理」か「固有の ID で管理」か、という違いです。 ID 方式はリンクが切れにくい反面、URL を見ても中身や置き場所が分かりにくいという側面もあります。

4クラウドドライブのアクセス権（共有）

クラウドにあるファイルは、「誰が」「何をできるか」を持ち主（オーナー）が決めます。 これを アクセス権（共有設定）と呼びます。2 つの軸で考えると整理できます。

軸その1：誰が開けるか（公開範囲）

  • 特定の人だけ：招待したメールアドレスの人だけ。いちばん安全。
  • 組織内の全員：例「名城大学のアカウントを持つ人なら開ける」。
  • リンクを知っている全員：URL さえあれば誰でも開ける。手軽だが拡散に注意。

軸その2：何ができるか（権限の強さ）

  • 閲覧者：見るだけ。
  • 閲覧者（コメント可）：見て、コメントを付けられる。
  • 編集者：中身を書き換えられる。
⚠ 「リンクを知っている全員」の落とし穴: 「リンクを知っている全員が編集者」にすると、URL が転送・公開されただけで、誰でも書き換え・削除できる状態になります。 個人情報・成績・未公開の作品などは、原則「特定の人だけ」にしましょう。
※ 攻撃者にパスワードを破られなくても、リンクが広まるだけで情報が漏れる――これも立派なリスクです。
📎 身近な実例: この講義資料（§1 の Drive リンク）は 「名城大学のアカウントを持つ人だけ」に設定しています。 だから個人の Google アカウントでは開けません。これがまさにアクセス権の制御の例です。

※ 共有メニューの場所や項目名はサービスの更新で変わります。迷ったら Google ドライブ公式ヘルプ「ファイルやフォルダを共有する」 や Google ドライブ 共有 権限 で検索を。

5GitHub ってなに？（概要）

GitHub（ギットハブ）（github.com）は、 プログラムのソースコードや文書を クラウドで保存・公開・共同編集できるサービスです。 世界中の開発者が日常的に使っています。この授業では、これから作る成果物を インターネットに公開する場所として使います。

  • リポジトリ（repository）：プロジェクト 1 つ分の保管場所（フォルダのようなもの）。 公開（public）と 非公開（private）を選べる。
  • GitHub Pages：リポジトリの中身を Webページとして無料で公開できる機能。次回 CL-12 で実際に使います。
  • Git：ファイルの「変更の履歴」を記録する仕組み。GitHub はその Git をクラウドで使える場所です（履歴管理は CL-14 で少し体験します）。
🔗 §4 とのつながり: 公開（public）リポジトリは世界中の誰でも見られます。クラウドドライブの「リンクを知っている全員」と同じ考え方です。 公開する場所には、人に見られて困るもの（個人情報・他人の著作物など）を置かないようにします。

5-1. 公開すると何が見える？（public と private）

GitHub Pages で Web ページを公開するとき、リポジトリの種類によって 「ソースコードまで見えるか」が変わります。

public（公開）リポジトリ

  • Web ページが誰でも見られる。
  • 中のソースコード（HTML や画像ファイルなど）も誰でも見られる。

private（非公開）リポジトリ

  • ソースコードは他の人から見えない。
  • ただし GitHub Pages で公開した Web ページ自体は、誰でも見られる。
🧩 もっと柔軟に公開を制御したいとき: 「特定の人だけに見せる」「独自ドメインで細かく設定する」など、より柔軟な公開設定が必要な場合は、 Cloudflare Pages などの別サービスも選択肢になります（この授業では扱いません）。

6GitHub アカウントを作る・安全に設定する

次回 CL-12 で GitHub を使えるよう、アカウントを作成し、 乗っ取られないための設定までやっておきます。 （細かいボタンの場所は変わるので、ここでは「何を・なぜ」を示します。手順は公式ドキュメントで確認しましょう。）

6-1. アカウントを作る

github.com の Sign up（新規登録） から登録します。登録時の選択で、特に次の 2 つを意識してください。

ユーザー名（公開されます）

  • ユーザー名は URL やページに表示され、世界中から見えます。
  • 本名や学籍番号をそのまま使わず、自分で決めた「ハンドル名」にするのがおすすめ（プライバシー保護）。

登録するメールアドレス（自分で選ぶ）

個人のメールでも大学のメールでも作れます。長く使い続けられるものを自分で選びましょう。

個人のメール

  • 卒業後もそのまま使い続けられる。
  • 普段使うアドレスを忘れにくい。

大学のメール

  • 在学中は管理しやすい。
  • 卒業すると使えなくなるため、その前にメール変更が必要。

本人確認

ユーザー名・メール・パスワードを入力したあと、「人間かどうか」の確認（パズルなど）と、 登録メールに届く確認コードの入力があります。コードを入れると登録完了です。

6-2. 乗っ取り防止の設定（重要）

GitHub のアカウントは、後から大切な成果物が貯まっていきます。最初に守りを固めておきましょう。

  1. 強いパスワードにする：長く、他のサービスと 使い回さない。 （使い回すと、別サービスから漏れたパスワードでログインを試される＝パスワードリスト攻撃に弱い。）
  2. 二要素認証（2FA）を設定する：ログイン時に、パスワードに加えて 認証アプリのコードや パスキーを要求する仕組み。
    → パスワードが漏れても、本人以外はログインできない。GitHub は 2FA の設定を強く推奨しています。
  3. リカバリコードを保存する：2FA を設定すると、使い捨ての リカバリコードが表示される。 スマホを失くしたときの「合鍵」なので、安全な場所に必ず保存（印刷・パスワード管理アプリなど）。
🔎 設定方法の調べ方: GitHub Docs「2 要素認証 (2FA) でアカウントを保護する」 ／ 検索なら GitHub 2要素認証 設定。画面の指示どおり進めれば完了します。
⚠ 認証アプリの端末を機種変更するとき: 2FA を設定したスマホを初期化・買い替えすると、ログインできなくなることがあります。 リカバリコードを保管しておけば復旧できます。捨てないように。

6-3. 個人のメールアドレスを隠す

GitHub では、操作の記録などに 登録メールアドレスが残って公開されてしまうことがあります。 個人のメールを知られたくない場合は、次の設定をしておきましょう（場所：Settings → Emails）。

  1. メールを非公開にする：Keep my email addresses private を ON。 これで GitHub があなたの代わりの noreply（返信不可）アドレスを用意してくれます。
  2. うっかり公開を防ぐ：Block command line pushes that expose my email を ON。 コマンドラインから操作したときに、メールが見えてしまう操作をブロックします。
  3. （コマンドラインで Git を使う人向け） Git に記録される送信元メールを、上記の noreply アドレスに変えておく： 
    git config --global user.email "12345678+username@users.noreply.github.com"
    ※ あなた専用の正確な noreply アドレスは Settings → Emails に表示されます。 その文字列をそのまま使ってください（12345678 や username は人によって違います）。
🔎 調べ方: GitHub Docs「コミットメールアドレスを設定する」 ／ 検索なら GitHub メールアドレス 非公開。
準備（提出は不要）

次回 CL-12 までに：GitHub アカウントを使える状態に

  • アカウントを作成し、2FA（二要素認証）まで設定しておく。
  • リカバリコードを保存しておく。
  • 個人のメールを 非公開に設定しておく（§6-3）。
